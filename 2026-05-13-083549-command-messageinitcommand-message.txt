⏺ I've read the game end to end — the welcome/level/session/game/summary
  screens, the session-builder logic, the answer/feedback flow, and the 11
  question banks. Here are concrete improvements, ordered by impact.

  Bugs and content defects

  1. Three exact-duplicate questions — e090 == e098, e091 == e099, e092 == e100
  (same text, answer, visual). Generator bug in the easy bank. Fix the band-3
  token-table template so it produces 11 distinct combinations, or trim the
  bank.
  2. attemptedCount counts submit clicks, not questions — submitAnswer
  increments it on every wrong submit and skipQuestion increments it once more.
  A kid who answers wrong twice then skips logs 3 "attempts" and 0 correct (0%
  accuracy) for a single question. Fix: increment attemptedCount once per
  distinct question reached, and track "first-try correct" separately for the
  accuracy display. Today's "Accuracy" stat is noise.
  3. Streak inflates on assisted answers — streak += 1 fires on any correct
  submit, including the one right after the explain overlay revealed the answer
  in bold. The "longest streak" stat doesn't measure independent solving. Fix:
  only increment streak when wrongSubmitsOnQuestion === 0 at submit time; reset
  it on hint/explain. The "Personal best" badge will then mean something.
  4. Medium band-2 explanations are placeholders — m035–m045, plus m080/m085 and
   others have explain: "✨ The answer is **N**." with no reasoning shown.
  Compare with the well-formed 🍊 42 − 18 = **24** left. For "change unknown"
  stories this is the most teachable moment — show the rewrite. Example for
  m035: 46 − 19 = 27 (work backward to find the start). Same pattern for m080 /
  m085.
  5. Massive hint/text uniformity inside a level — out of 678 records, only 107
  unique hints. e037–e069 (32 questions) share the exact text template "A baker
  set out N plain bagels and M raisin bagels…" / "The relay team passed the
  baton N times…" / "The museum cart…" and the identical hint "Combine both
  groups from the story." Kids will see the same hint repeated 6–8 times in a
  20-question medium run. The Python generators (gen_banks.py,
  gen_extended_banks.py) produced this — they should rotate hint phrasings per
  question and vary surface stories more aggressively.
  6. Skill badges may be unreachable in 20-question sessions — maxPerType = 
  Math.max(2, Math.floor(want * 0.35)) (= 7 at want=20) and needBadge=3 at
  sessionQueue<25. With band quotas {1:7, 2:8, 3:5}, several types compete for
  slots; some sessions will produce 0–2 of "Equal Groups" or "Sharing" and the
  kid never hits the threshold even with perfect play. Worth printing "2 of 3
  toward Sharing badge" in the summary so the system feels legible.

  UX

  7. 2-second auto-advance is too fast for some readers —
  setTimeout(goNextQuestion, 2000) after a correct answer. The banner says
  "Press Enter or wait — next story is on the way!" but a slow reader or
  screen-reader user gets cut off. Honor prefers-reduced-motion by removing the
  timer (require Enter / a "Next" button), and consider making it a setting.
  8. No way to reset the local save — wordProblemAdventure_v1 persists
  bestStreakEver, recentQuestionIds, lastLevel. A kid who wants a fresh streak
  target can't reset without DevTools. Add a small "Reset progress" link on the
  summary screen behind a confirm prompt.
  9. Doesn't tell the hub anything beyond "played" — the game posts played +
  celebrate on summary visibility (line 2403) but never
  KLS.bridge.session(level, {questionsAnswered, accuracy, bestStreak}) or
  KLS.bridge.stars(level, n). The hub already supports it
  (progress.recordSession and progress.awardStars) and would surface
  accuracy/streak on the tile. Wire up: when endSession fires, also post a
  session message with attemptedCount, correctCount/attemptedCount, and
  sessionBestStreak, and an awardStars mapped from accuracy thresholds (e.g.
  ≥90%/≥75%/≥50% → 3/2/1 stars).
  10. Picture toggle isn't a real toggle for assistive tech — #btn-picture flips
   its label but has no aria-pressed. Add aria-pressed="true|false" and remove
  the hand-rolled .easy-visible class hack
  (.btn.hide-easy-only:not(.easy-visible) { display: none } — just set hidden
  directly).
  11. Sound has no off switch — playCorrectSound/playWrongSound fire on every
  submit. Add a mute toggle in the chrome strip (persist in storage). Tutoring
  sessions in libraries/classrooms will appreciate it.
  12. Submit on wrong sticks the keyboard focus — submitAnswer doesn't refocus
  the input after showing the hint. Kids on iPad get bumped out of the numeric
  keypad and have to tap again. Add input.focus() (and input.select()) after
  showFeedbackHint/showFeedbackExplain.

  Code quality

  13. inferProblemType is mostly dead code — almost every question now has an
  explicit problemType. The fallback regex chain (lines 1540–1585) is brittle
  (the medium branch silently defaults to "Separate"; the hard branch defaults
  to "Multi-step" even when nothing matches). Either delete it and make
  problemType required in the schema (the Python validator should enforce it),
  or keep only as a debug-time assertion.
  14. Question banks belong in a separate file — 700+ inline records inside
  index.html make the file 2410 lines, hostile to diff/review and to bank
  regeneration. Move them to questions/easy.json, questions/medium.json,
  questions/hard.json and load via <script src> (assignment to globals) so it
  stays file:// compatible. The Python generators can then write JSON directly
  instead of regex-patching HTML.
  15. recordShownQuestionId is called inside renderQuestion — meaning a question
   is marked "shown" the moment the screen renders, even if the kid immediately
  exits. That's mostly fine, but the function also reads, mutates, and writes
  the whole wordProblemAdventure_v1 blob on every render. Cheap, but consider
  debouncing or batching at session end.
  16. enrichQuestion always rebuilds the object — Object.assign({}, item, { 
  problemType: ... }) runs on every question every time buildSessionQueue runs.
  Since problemType is already set on virtually every record, this is wasted
  work. Once you delete inferProblemType, this whole function disappears.
  17. finalKidFriendlyOrder can fall back through buckets with 
  lastBand-violations — its fallback line chosen = pullFrom(b2) || pullFrom(b1) 
  || pullFrom(b3); ignores the "don't repeat lastBand" rule under shortage. Not
  a bug per se, just worth a comment so a future reader knows the constraint is
  best-effort.

❯ I want to allow kids to set up account with funny mojis, keep track of their  
  learning status for each games.                                               
  The kid picks an emoji and a nickname on first visit; everything lives in the 
   browser.                                                                     
  It uses Device-local profile.                                                 
                                                                                
  In addition, I like to include the following features.                        
  1) Auto-save + explicit "save & quit" — kids close tabs unpredictably.        
  Auto-save at checkpoints, plus a visible save button that confirms with a     
  tiny animation. Don't rely on the kid pressing it.                            
                                                                                
  2) Opaque per-game state blob — different games have wildly different state   
  shapes (story-problem vs. matching vs. puzzle). Each game owns its state      
  shape as JSON; the store just persists it. Future-proofs you against game     
  changes.                                                                      
                                                                                
  3) Schema versioning from day 1 — Build the migration scaffolding now, even   
  if v1 is empty. Costs nothing, saves a future production migration.           
                                                                                
  4) Export / Import a profile to a JSON file — the killer feature for a        
  no-backend design. The kid (or parent) can download their save, take it to a  
  different device, or restore after a browser clear. Replaces ~80% of the      
  value of an account with a one-click download.                                
                                                                                
  5) Resume modal on game entry — if saved state exists for (profile, game),    
  prompt "Continue where you left off?" vs. "Start over." Standard              
  game-industry pattern; kids understand it.                                    
                                                                                
  6) Profile management UI — edit emoji/nickname, delete profile, cap at 6.     
  Without this, your test data is forever.                                      
                                                                                
  7) "Last played" + progress badge on the game hub tiles — the visible reward
